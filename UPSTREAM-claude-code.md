## Feature Requests

- **LSP `extensionToLanguage` cannot be dynamic, computed, or overridden per project** (2026-07-11) — A plugin's `lspServers.<id>.extensionToLanguage` must be a **static** map. There are no wildcards or globs, no way to point `lspServers` at an executable whose stdout is the config, no lifecycle hook that runs before `lspServers` is parsed, and no way for a project's `.claude/settings.json` to override a plugin-provided server's extension map. Only three interpolations exist (`${CLAUDE_PLUGIN_ROOT}`, `${CLAUDE_PLUGIN_DATA}`, `${CLAUDE_PROJECT_DIR}`) and they cannot appear in the map's keys.
  This is a hard bottleneck for **polyglot** language servers. ast-grep lints 28 languages, but *which* ones matter is a property of the individual project — it is exactly the set of `language:` values in that project's own `.ast-grep/rules/*.yml`, and it is trivially derivable from them. There is no way to express that. Worse, a static map can never even be **complete**: ast-grep's `customLanguages` lets a project register a tree-sitter grammar under **arbitrary** extensions that no plugin-side map can anticipate. Upstream's own VSCode extension sidesteps this entirely by claiming every file (`documentSelector: [{ scheme: 'file', language: '*' }]`) and gating on the config file's presence instead — an option Claude Code does not offer.
  Ownership: upstream · Workaround: partial — ship a conservative static map and accept that it is wrong for somebody.

- **No LSP extension-collision precedence, and no documented fallthrough** (2026-07-11) — When two enabled LSP servers declare the same extension in `extensionToLanguage`, "the first server registered handles files with that extension and the others never start". But **registration order is undocumented** (alphabetical? install order? settings order?), there is **no priority/ordering field**, and there is **no way to run both** servers on one extension. So a lint-shaped server and a types-shaped server cannot coexist on `.ts` at all, and which one wins is unpredictable.
  The load-bearing undocumented case: **if the first claimer dies during `initialize`, is the extension freed for the next claimer, or starved?** `ast-grep lsp` exits during initialize in any project lacking an `sgconfig.yml` — i.e. most projects. If Claude Code falls through, a polyglot lint server could safely claim broadly and simply be inert where it has no config. If it starves, the same claim silently disables `pyright-lsp`/`rust-analyzer-lsp` in every repo. **The answer completely determines a safe plugin design, and it is not documented.** We assumed the pessimistic reading and claim only what we lint.
  Ownership: upstream · Workaround: partial — never claim an extension an official language server owns; enforced by a check in this repo.

- **No LSP client-side file watching (`workspace/didChangeWatchedFiles`)** (2026-07-11) — Claude Code does not implement file watching on behalf of a language server, so a server that needs to know when a non-open file changed (its own config, a lockfile, a rule set) cannot find out. **This was previously filed here as a Bug. It is not one, and the correction matters** — see below.
  **MEASURED (2026-07-11): Claude Code advertises `workspace.didChangeWatchedFiles.dynamicRegistration = undefined`.** Taken by logging the client's `initialize` from inside the server, in a real session (`claude --plugin-dir … --debug-file`). No public dump of Claude Code's LSP capabilities existed before this.
  **That makes Claude Code spec-compliant here, not broken.** LSP 3.17, *Register Capability*, verbatim: *"Not all clients need to support dynamic capability registration. **A client opts in via the `dynamicRegistration` property on the specific client capabilities.**"* Claude Code never opts in, so it is never obliged to honour a registration — and a client cannot break a promise it never made. Answering `-32601 MethodNotFound` for a method it does not implement is defensible JSON-RPC. **The fault in the ast-grep case is ast-grep's** (it registers without checking the capability, which the same spec section forbids — see `UPSTREAM-brew--ast-grep.md`).
  Consequently **`rust-analyzer`, `pyright`, `gopls` and `clangd` are NOT broken here** — they respect the capability, decline to register, and self-watch. (An earlier revision of this entry speculated they were silently broken. They are not. The correction is recorded rather than deleted, because a wrong note compounds.)
  So this is a **missing feature, not a violation**: servers that respect the capability degrade correctly, and only servers that ignore it get hurt. It still costs real functionality — every server that depends on watched-file notifications loses config/rule hot-reload — and a client that watched files would fix the whole class at once, including for spec-violating servers.
  `anthropics/claude-code#32595` and its re-file `#52693` are both **closed as NOT_PLANNED**. A third-party byte-log in `anthropics/claude-plugins-official#1359` shows `workspace/configuration` and `window/workDoneProgress/create` getting the same `-32601`, so the gap is broader than watched files.
  ⚠️ *Unverified, carried from earlier research and NOT re-confirmed here:* that `csharp-lsp` and `elixir-lsp` **hang** on the unanswered request. If Claude Code really answers `-32601`, a blocking server would receive an error response and unblock, so this claim is suspect. Flagged rather than repeated as fact.
  Ownership: upstream · Workaround: **full** — `bin/lsp-shim.mjs` (0.4.0) answers the registration on the client's behalf, watches the globs the server asked for, and injects the notification. It also **injects `dynamicRegistration: true` into the forwarded `initialize`** — which is the honest statement, since with the shim in place the client-side really does implement it, and it is what keeps the shim working if ast-grep ever fixes its own compliance.
