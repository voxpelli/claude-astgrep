## Feature Requests

- **LSP `extensionToLanguage` cannot be dynamic, computed, or overridden per project** (2026-07-11) — A plugin's `lspServers.<id>.extensionToLanguage` must be a **static** map. There are no wildcards or globs, no way to point `lspServers` at an executable whose stdout is the config, no lifecycle hook that runs before `lspServers` is parsed, and no way for a project's `.claude/settings.json` to override a plugin-provided server's extension map. Only three interpolations exist (`${CLAUDE_PLUGIN_ROOT}`, `${CLAUDE_PLUGIN_DATA}`, `${CLAUDE_PROJECT_DIR}`) and they cannot appear in the map's keys.
  This is a hard bottleneck for **polyglot** language servers. ast-grep lints 28 languages, but *which* ones matter is a property of the individual project — it is exactly the set of `language:` values in that project's own `.ast-grep/rules/*.yml`, and it is trivially derivable from them. There is no way to express that. Worse, a static map can never even be **complete**: ast-grep's `customLanguages` lets a project register a tree-sitter grammar under **arbitrary** extensions that no plugin-side map can anticipate. Upstream's own VSCode extension sidesteps this entirely by claiming every file (`documentSelector: [{ scheme: 'file', language: '*' }]`) and gating on the config file's presence instead — an option Claude Code does not offer.
  Ownership: upstream · Workaround: partial — ship a conservative static map and accept that it is wrong for somebody.

- **No LSP extension-collision precedence, and no documented fallthrough** (2026-07-11) — When two enabled LSP servers declare the same extension in `extensionToLanguage`, "the first server registered handles files with that extension and the others never start". But **registration order is undocumented** (alphabetical? install order? settings order?), there is **no priority/ordering field**, and there is **no way to run both** servers on one extension. So a lint-shaped server and a types-shaped server cannot coexist on `.ts` at all, and which one wins is unpredictable.
  **The load-bearing question — "if the first claimer dies, is the extension freed or starved?" — is now ANSWERED, and it splits three ways.** We originally wrote that it was undocumented and assumed the pessimistic reading. Partly wrong, partly right, and the distinction matters:
  1. **Registration-time config failure** (missing `command`/`extensionToLanguage`): **FIXED in v2.1.205** — *"Fixed a plugin LSP server that fails to initialize preventing a valid LSP server from another plugin handling the same file extension."* The bad server never claims the slot. Documented in `plugins-reference`: *"Before v2.1.205, a server that failed to initialize still claimed its extensions and blocked another valid server."*
  2. **Registers successfully, then dies** (spawn failure, `initialize` handshake, `startupTimeout`, crash): **STILL STARVED.** The binary's router does `let b = S[0]` — always index 0, with no failover to `S[1]`. It retries up to `maxRestarts`; it never yields the extension. **This is exactly ast-grep's case** (`ast-grep lsp` exits before the LSP handshake in any project without an `sgconfig.yml` — i.e. most projects), so the conservative claim rule **stays necessary**.
  3. **Two healthy servers, same extension:** first-registered wins permanently; the runner-up never starts. By design, unchanged (#32912, #58332 closed won't-do).
  So our narrow extension claim is still correct — but for reason (2), not the vaguer "it's undocumented" we first wrote down. The remaining asks: a documented ordering/priority field, and a failover when a registered server dies.
  Ownership: upstream · Workaround: partial — never claim an extension an official language server owns; enforced by a check in this repo.

- **LSP client-side file watching (`workspace/didChangeWatchedFiles`)** (2026-07-11) — Claude Code doesn't watch files on a language server's behalf, so a server that needs to know when a *non-open* file changed — its own config, a lockfile, a rule set — has no way to find out. It would be a genuinely useful capability to add.
  **Measured (2026-07-11): Claude Code advertises `workspace.didChangeWatchedFiles.dynamicRegistration = undefined`.** Taken by logging the client's `initialize` from inside the server in a real session (`claude --plugin-dir … --debug-file`). As far as we can tell there was no public dump of Claude Code's LSP capabilities before this.
  **To be clear, that's the correct thing to advertise, and this was originally — and wrongly — filed here as a Bug.** LSP 3.17 makes dynamic registration opt-in: *"Not all clients need to support dynamic capability registration. A client opts in via the `dynamicRegistration` property on the specific client capabilities."* Claude Code doesn't opt in, so it was never obliged to honour a registration, and answering `-32601` for a method it doesn't implement is reasonable JSON-RPC. Not implementing something you never claimed isn't a defect — it's a gap, which is why this now sits under Feature Requests.
  **The impact is wider than "servers that misbehave get hurt", and an earlier revision of this entry got it backwards twice.** Only **rust-analyzer** actually self-watches when the capability is absent (`crates/rust-analyzer/src/config.rs` gates its `Client` default on `did_change_watched_files_dynamic_registration()` and otherwise drops to `FilesWatcher::Server`). **`pyright` does NOT** — its LSP server checks the capability, correctly declines to register, and then has no fallback at all (the real OS-level watcher is wired only into the standalone CLI, never `server.ts`). So pyright silently misses on-disk changes here too. **gopls** shipped a server-side watcher in v0.22.0, but it defaults to `off` and its source carries a literal `// TODO: support "auto" mode` — the automatic trigger doesn't exist yet. So *most* servers lose config/rule hot-reload in Claude Code, quietly. Implementing the capability would fix the whole class at once.
  Context: `anthropics/claude-code#32595` and its re-file `#52693` are both closed as NOT_PLANNED. `anthropics/claude-code#16360` (OPEN) reports the same `-32601` on `workspace/configuration` and `window/workDoneProgress/create`, and notes the client advertises `workspace.configuration: false` — structurally the same shape as the finding here.
  ❌ **RETRACTED — the "servers hang" claim was folklore, and we were repeating it.** Earlier revisions said `csharp-lsp` and `elixir-lsp` **hang** on an unanswered request, and that both ship in "Anthropic's own marketplace". Both parts are wrong. **(a)** The request IS answered (`-32601`, measured on the wire at 2.1.207), so a blocking server unblocks. The "hangs forever" language traces to two 2026-03/04-era reports (#32595, #52693) that inferred silence from symptoms **without a byte trace**, and was then propagated verbatim by a downstream plugin README. No byte-level evidence of a silent hang exists on any version. **(b)** `elixir-lsp` is **not** an Anthropic plugin — it is third-party (`georgeguimaraes/claude-code-elixir`); the official marketplace ships 13 `*-lsp` plugins and no Elixir. The real failure mode is server-side handling of the *error response*: csharp-ls aborts its solution load on a later unguarded `window/workDoneProgress/create`, Roslyn/StreamJsonRpc throws `RemoteMethodNotFoundException`, PowerShell Editor Services times out after ~30s. Not hangs.
  Ownership: upstream · Workaround: **full** — `bin/lsp-shim.mjs` answers the registration on the client's behalf, watches the globs the server asked for, and injects the notification. It also injects `dynamicRegistration: true` into the forwarded `initialize`, which is an honest declaration — with the shim in place, the client side genuinely does implement it.

## Bugs

- **The `window/*` server→client NOTIFICATION channel is unimplemented — `window/logMessage` and `window/showMessage` are discarded, so LSP servers are undebuggable** (2026-07-11) \[degraded] — Claude Code surfaces a language server's **stderr** (as `[LSP SERVER …]` in `--debug-file`), but silently drops the server's protocol-level messages. A server that reports a problem the way the protocol says to is reporting into a void.
  **Confirmed at source, not just empirically** (Claude Code 2.1.207 is a Bun single-file executable with the JS embedded and greppable). It registers exactly **one** notification handler and **one** request handler on a plugin LSP server:
  ```
  onNotification("textDocument/publishDiagnostics")   <- the only one
  onRequest("workspace/configuration")                <- the only one
  ```
  `window/logMessage` and `window/showMessage` appear in the bundle **only as vendored `vscode-languageserver-protocol` constants — there is no handler**. The bundled `vscode-jsonrpc` routes unhandled notifications to an `unhandledNotificationEmitter` that Claude Code never subscribes to. **Dispatch is by method name, before params are ever read**, so a severity filter is structurally impossible, and there is no interactive-vs-headless difference to hide behind — it is one binary with no code path to a toast, a status line, or `/doctor`.
  **This is why nobody has reported it.** The `window/*` surface splits in two, and only one half is visible: server→client **requests** (`window/workDoneProgress/create`, `client/registerCapability`, `workspace/configuration`) carry an `id`, get a loud `-32601`, and are already filed (#16360 open, plugins-official#1359 open, #18941). Server→client **notifications** carry no `id`, so declining them produces **no artifact at all**. There is nothing to screenshot and nothing to grep. You only find it by forcing a server to emit an ERROR-level `logMessage` and proving its absence from a debug log that faithfully records the surrounding protocol traffic.
  **This is the one that costs real time, and it's worth fixing because it hides everything else.** We spent the better part of a day working out why ast-grep had stopped reloading its rules — waiting on suspected LSP lag, touching files, diffing a stale diagnostic message against freshly-correct line numbers. **The server had been telling us the whole time.** ast-grep detects the failed registration and reports it accurately, naming the exact cause; the message just never reaches a human.
  **Reproduction** (drive `ast-grep lsp` over stdio and reply `-32601` to `client/registerCapability`, as Claude Code does — the server immediately says):
  ```
  [window/logMessage ERROR] Failed to register file watchers:
      Error { code: MethodNotFound, message: "Unhandled method client/registerCapability", data: None }
  ```
  **And in a real Claude Code session, that channel is empty:** 72 KB of `--debug-file` output from a run with `ast-grep lsp` attached contains **zero** `window/logMessage` entries — not the `ERROR` above, and not even the routine `INFO` ones the same server emits on every startup (`"server initialized!"`, `"Starting rule reload..."`, `"Rules reloaded successfully"`). The messages are produced; they do not arrive.
  **`window/showMessage` is dropped too — and that is the sharper half of this.** `showMessage` is the method LSP defines specifically for *"show this to the user"*; it is not a log channel a client may reasonably ignore. Concrete, reproduced consequence: save a syntactically invalid ast-grep rule and the server emits **both** a `showMessage` \[ERROR] and a `logMessage` \[ERROR] (`Failed to load rules: Cannot parse rule …`) and keeps the last-known-good ruleset. In Claude Code the user sees **neither** — their diagnostics simply change and nothing explains why. **We spent a day blaming ast-grep for that silence** (and nearly filed it as `ast-grep#722`). The silence is the client's.
  Impact is not specific to ast-grep. `window/logMessage` / `window/showMessage` are *the* standard way a language server reports a missing toolchain, an unreadable config, a failed index. Dropping both means **every** LSP failure in Claude Code presents as unexplained silence, and every user must reverse-engineer what the server would simply have told them — while the docs offer no alternative: `plugins-reference` documents no client-capability list, no server→client method handling, and **no LSP log surface at all** (`--debug` is scoped to *plugin loading*; `--debug-file` is never mentioned in an LSP context). A server that logs correctly per spec is undebuggable by design.
  Ask, in ascending order of effort: (1) surface `window/logMessage` in `--debug` next to the stderr that already appears there; (2) surface `window/showMessage` to the user, as the method exists for; (3) document that neither is currently handled, so server authors know.
  Severity: degraded · Ownership: upstream · Workaround: **partial** — read the server's *stderr* via `claude --debug-file`, which *is* surfaced. That only helps for servers that also log to stderr; a server reporting exclusively over `window/*`, exactly as the protocol invites, is invisible.

## Upstream Opportunities

- **A stdio shim that answers `client/registerCapability` on the client's behalf** (2026-07-11) — Exists only because the client will not. **Explicitly a stopgap, not a product** — see the retirement trigger below. Recorded here because it is a working existence proof, not because it wants to be a plugin.
  Its general use is that answering the registration properly sidesteps a whole class of downstream damage: the `-32601` is handled gracefully by ast-grep but not by everyone — csharp-ls aborts its solution load on a later unguarded request, Roslyn/StreamJsonRpc throws, and Kate's identical `-32601` crashes phpactor's RPC layer outright (`phpactor#1304`). If anyone wants to argue #32595 back open, a shim demonstrates the mechanism works and costs upstream nothing to verify. (Prior art exists and we were not first: `georgeguimaraes/claude-code-elixir` ships a Python wrapper that intercepts the same request, and a community `powershell-lsp` plugin does the *inverse* — it strips `dynamicRegistration` from the forwarded `initialize` so the server never asks. Both are reasonable; ours differs in that it *implements* the capability rather than suppressing it.)
  **Retirement trigger: the ast-grep self-watch fallback** (see `UPSTREAM-brew--ast-grep.md`). That fix is entirely server-side, needs nothing from Anthropic, and makes this shim dead code for our purposes. It is the durable fix; this is the bridge to it. **Not filed — tracked here only.**
  Source: `bin/lsp-shim.mjs`, `bin/glob-watch.mjs`, `bin/lsp-framing.mjs` (this repo, 0.4.0) · Merge readiness: proof-of-concept
  Ownership: us · Workaround: full — shipped in this plugin, pending the upstream fix that would retire it.
