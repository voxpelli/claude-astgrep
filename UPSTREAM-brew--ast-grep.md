## Feature Requests

- **A self-watch fallback for clients that don't provide file watching** (2026-07-11) — ast-grep's language server asks the *client* to watch `**/*.{yml,yaml}` for it, via a `client/registerCapability` request, and reloads its rules when the client reports a change. That is a perfectly reasonable design, it is well within the spec, and it works beautifully in VSCode. It just leaves hot-reload dependent on the client — and some clients don't implement file watching at all.
  **Claude Code is one of them.** It advertises `workspace.didChangeWatchedFiles.dynamicRegistration` as `undefined` (measured 2026-07-11, by logging the client's `initialize` from inside the server) and declines the registration. ast-grep then never hears about a rule change and serves its startup rule set for the life of the session. Everything else keeps working perfectly — document sync, re-analysis, diagnostics — so from a user's seat it just looks like their *rule* is wrong.
  **The suggestion: when the client doesn't advertise that capability, watch the rule files in-process instead.** The signal is already sitting in `initialize`; it just isn't consulted today. There's good precedent for exactly this — **rust-analyzer** ships it as `files.watcher: "client" | "server"` (server-side via `notify`), and **pyright** runs its own watchers. Both are consequently unaffected by this whole class of problem, in Claude Code and anywhere else.
  **This is not a request to reverse #2142.** The maintainer's reasoning there (closed, shipped 0.39.4) — *"I found LSP should ask Client to provide file watch and reload config. So this should be done in this repo instead of vscode or other clients."* — is sound, and asking the client first should stay the default. The ask is only for a fallback when the client says it can't help. Small change, entirely server-side, needs nothing from Anthropic, and it would let downstream shims like ours be deleted.
  Related: **#722** (open, *"LSP: inform user when rule is invalid"*) interacts with this — on reload an invalid rule is silently ignored, so a torn read of a half-written file can make diagnostics quietly disappear.
  Ownership: upstream · Workaround: **full** — `bin/lsp-shim.mjs` in this repo answers the registration on the client's behalf, watches the globs the server asked for, and injects `workspace/didChangeWatchedFiles`. Verified end-to-end by `verify-lsp-reload.mjs`.

- **A no-op mode when no `sgconfig.yml` is present** (2026-07-11) \[minor] — The server exits during `initialize` with `Error: No ast-grep project configuration is found.` (exit code 3). Entirely sensible on its own — but it means an editor integration that registers the server **globally** ends up with a dead server in every project that doesn't use ast-grep, which is most of them. The VSCode extension sidesteps this by gating activation on `findConfigFile()`; a client whose plugin model has no activation condition (Claude Code) can't.
  A `--no-config-ok` flag — start, report zero rules, publish nothing — would let integrations register the server broadly and safely.
  Two smaller notes in the same area, in case they're useful: LSP has a defined shape for this (fail the `initialize` request with an `InitializeError`, letting the client decide whether to retry or surface it), which would give clients something structured to show a user instead of a process that vanished mid-handshake. And in a client where the first server claiming a file extension wins and the others never start, a dead-on-arrival server could — depending on unspecified client behaviour — starve that extension for the real language server, which is what pushed us to a defensively narrow extension claim.
  Severity: minor · Ownership: upstream · Workaround: full — pass `-c <path>` explicitly, and document that the plugin is a no-op without a config.

## Bugs

- **The server re-detects language from the file path and ignores the client's `languageId`** (2026-07-11) \[minor] — A buffer opened with `languageId: "javascript"` at a path ast-grep maps differently gets ast-grep's answer rather than the client's. Only the *keys* of an `extensionToLanguage`-style map end up mattering; the values are cosmetic.
  Mostly benign and arguably pragmatic, but it does mean the server can't lint an unsaved or virtual buffer whose language is known only from `languageId`, and it quietly differs from other language servers on a detail clients are entitled to rely on. Worth documenting even if it isn't changed.
  Severity: minor · Ownership: upstream · Workaround: full — name fixture files with the extension the rule's `language:` implies.

## Notes (not defects — recorded so we don't re-litigate them)

- **ast-grep registers its watchers without first checking `didChangeWatchedFiles.dynamicRegistration`.** We initially wrote this up as a spec violation. **It isn't, and the retraction is worth keeping.** LSP 3.17 says only *"Not all clients need to support dynamic capability registration. A client opts in via the `dynamicRegistration` property"* — descriptive, no MUST or SHOULD. The one imperative nearby (*"if a server wants to support both static and dynamic registration it needs to check the client capability…"*) is scoped to servers offering **both**, to stop them double-registering, and can't apply here at all: `DidChangeWatchedFiles` has **no static path** (*"the current protocol doesn't support static configuration for file changes from the server side"*). Asking, and being told no, is a legitimate outcome. Checking the capability is worth doing only as part of the self-watch fallback above — on its own it would make ast-grep correctly *stop* asking, taking hot-reload from broken to impossible.
- **ast-grep reports the refused registration correctly — nobody sees it because the client drops the message.** Source (`crates/lsp/src/lib.rs`): `if let Err(e) = self.register_file_watchers().await { … log_message(MessageType::ERROR, "Failed to register file watchers: {e:?}") }`. Verified live: reply `-32601` and the server immediately says so, naming the failed method. Claude Code discards `window/logMessage` entirely, which is what turned a precise error report into an unexplained silence. That one is filed against the client (`UPSTREAM-claude-code.md`), not here — **ast-grep did its part.**

## Upstream Opportunities

- **An ast-grep language-server plugin for Claude Code** (2026-07-11) — Claude Code supports registering language servers from a plugin (`lspServers`), and `ast-grep lsp` works well with it: rules produce live diagnostics in-context after each edit. No such plugin existed, so we built one — including the extension-collision rule that makes it safe to install (never claim a file extension an official language server owns, because the first server registered for an extension wins and the others never start), and a zero-dependency stdio LSP verifier that asserts a diagnostic **fires** on a violating buffer and **clears** on the fixed one.
  Might suit ast-grep's own docs (`guide/tools/editors.html` lists nvim/coc/helix/emacs but not Claude Code), or its existing Claude Code plugin (see `UPSTREAM-plugin--ast-grep.md`).
  Source: `voxpelli/claude-astgrep` (this repo) · Merge readiness: direct
  Ownership: us · Workaround: full — published as a standalone plugin.
