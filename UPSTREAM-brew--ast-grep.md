## Feature Requests

- **A self-watch fallback for clients that don't provide file watching** (2026-07-11) — ast-grep's language server asks the *client* to watch `**/*.{yml,yaml}` for it, via a `client/registerCapability` request, and reloads its rules when the client reports a change. That is a perfectly reasonable design, it is well within the spec, and it works beautifully in VSCode. It just leaves hot-reload dependent on the client — and some clients don't implement file watching at all.
  **Claude Code is one of them.** It advertises `workspace.didChangeWatchedFiles.dynamicRegistration` as `undefined` (measured 2026-07-11, by logging the client's `initialize` from inside the server) and declines the registration. ast-grep then never hears about a rule change and serves its startup rule set for the life of the session. Everything else keeps working perfectly — document sync, re-analysis, diagnostics — so from a user's seat it just looks like their *rule* is wrong.
  **The suggestion: when the client doesn't advertise that capability, watch the rule files in-process instead.** The signal is already sitting in `initialize`; it just isn't consulted today.
  **Precedent — stated honestly, because we got this wrong once already.** **rust-analyzer does exactly this**, and it is clean: `crates/rust-analyzer/src/config.rs` gates its `files.watcher` default of `Client` on `did_change_watched_files_dynamic_registration()`, and otherwise falls through to `FilesWatcher::Server` (a real `notify`-backed watcher). Automatic, capability-driven, not a manual opt-in. **gopls** shipped a server-side watcher in v0.22.0 (May 2026) but it defaults to `off` and its source carries a literal `// TODO(hxjiang): support "auto" mode` — the maintainers are building toward the same behaviour and haven't automated the trigger yet. ⚠️ **We previously also cited pyright here. That was wrong** — pyright's LSP server checks the capability and correctly declines to register, but has **no fallback**: the real OS-level watcher (chokidar) is wired only into the standalone CLI, never `server.ts`. Pyright misses on-disk changes in such clients exactly as ast-grep does. So the honest claim is *"rust-analyzer already does this, and gopls is building toward it"* — one strong precedent plus a direction of travel, not an industry norm.
  **This is not a request to reverse #2142.** The maintainer's reasoning there (closed, shipped 0.39.4) — *"I found LSP should ask Client to provide file watch and reload config. So this should be done in this repo instead of vscode or other clients."* — is sound, and asking the client first should stay the default. The ask is only for a fallback when the client says it can't help. Entirely server-side, needs nothing from Anthropic, and it would let downstream shims like ours be deleted. (Cost is real but modest: `notify` is not currently a dependency — it would be a new one. It is the same crate rust-analyzer uses.)
  Ownership: upstream · Workaround: **full** — `bin/lsp-shim.mjs` in this repo answers the registration on the client's behalf, watches the globs the server asked for, and injects `workspace/didChangeWatchedFiles`. Verified end-to-end by `verify-lsp-reload.mjs`.

- **Escalate a failed watch registration to `window/showMessage`, not `window/logMessage` alone** (2026-07-11) \[minor] — When `client/registerCapability` is refused, ast-grep logs `MessageType::ERROR` via `window/logMessage` and carries on. That is correct, but `logMessage` is understood ecosystem-wide as *"goes to a log panel the user may never open"*. **rust-analyzer's own convention** (`show_and_log_error`, `crates/rust-analyzer/src/lsp/utils.rs`) is to emit **both** a log line *and* a `window/showMessage` for anything the user actually needs to see — sometimes a `ShowMessageRequest` with an "Open server logs" action.
  Losing hot-reload silently for an entire session qualifies. And this is the one improvement ast-grep can make **unilaterally**, without waiting on any client: it costs one extra line and does not depend on the fallback above landing. (It would not help in Claude Code specifically — which drops `showMessage` too — but it would help everywhere else, and it is right regardless.)
  Severity: minor · Ownership: upstream · Workaround: none needed.

- **A no-op mode when no `sgconfig.yml` is present** (2026-07-11) \[minor] — With no config, `ast-grep lsp` prints `Error: No ast-grep project configuration is found.` and exits **3**. Entirely sensible on its own — but it means an editor integration that registers the server **globally** gets a dead process in every project that doesn't use ast-grep, which is most of them. The VSCode extension sidesteps this by gating activation on `findConfigFile()`; a client whose plugin model has no activation condition (Claude Code) can't.
  A `--no-config-ok` flag — start, report zero rules, publish nothing — would let integrations register the server broadly and safely.
  **Precision, since we got this wrong once:** this is **not** a failure "during `initialize`", and so *"return an `InitializeError` instead"* is not a coherent ask. `main_with_args()` unwraps the project and calls `exit_with_error()` **before** `run_language_server()` — the process is gone before tower-lsp ever attaches to stdio. There is no JSON-RPC handshake at all, not even a failed one.
  **And the stakes here are real, not hypothetical.** In Claude Code, a server that registers for an extension and then dies **still starves that extension** — the router reads only index 0 and never fails over (v2.1.205 fixed only the *registration-time config* failure case, not this one). So a globally-registered `ast-grep lsp` that exits for lack of a config can silently deny `.js`/`.ts` to the real language server. That is precisely what forced our defensively narrow extension claim.
  Severity: minor · Ownership: upstream · Workaround: full — pass `-c <path>` explicitly, and document that the plugin is a no-op without a config.

## Bugs

- **The server re-detects language from the file path and ignores the client's `languageId`** (2026-07-11) \[minor] — A buffer opened with `languageId: "javascript"` at a path ast-grep maps differently gets ast-grep's answer rather than the client's. Only the *keys* of an `extensionToLanguage`-style map end up mattering; the values are cosmetic.
  Mostly benign and arguably pragmatic, but it does mean the server can't lint an unsaved or virtual buffer whose language is known only from `languageId`, and it quietly differs from other language servers on a detail clients are entitled to rely on. Worth documenting even if it isn't changed.
  Severity: minor · Ownership: upstream · Workaround: full — name fixture files with the extension the rule's `language:` implies.

## Notes (not defects — recorded so we don't re-litigate them)

- **ast-grep registers its watchers without first checking `didChangeWatchedFiles.dynamicRegistration`.** We initially wrote this up as a spec violation. **It isn't, and the retraction is worth keeping.** LSP 3.17 says only *"Not all clients need to support dynamic capability registration. A client opts in via the `dynamicRegistration` property"* — descriptive, no MUST or SHOULD. The one imperative nearby (*"if a server wants to support both static and dynamic registration it needs to check the client capability…"*) is scoped to servers offering **both**, to stop them double-registering, and can't apply here at all: `DidChangeWatchedFiles` has **no static path** (*"the current protocol doesn't support static configuration for file changes from the server side"*). Asking, and being told no, is a legitimate outcome. Checking the capability is worth doing only as part of the self-watch fallback above — on its own it would make ast-grep correctly *stop* asking, taking hot-reload from broken to impossible.
- **ast-grep reports the refused registration correctly — nobody sees it because the client drops the message.** Source (`crates/lsp/src/lib.rs`): `if let Err(e) = self.register_file_watchers().await { … log_message(MessageType::ERROR, "Failed to register file watchers: {e:?}") }`. Verified live, with causation isolated: drive the server with a client that **refuses** the registration (`-32601`) and the ERROR fires; drive it with one that **accepts**, and it doesn't. Claude Code registers no handler for `window/logMessage`, which is what turned a precise error report into an unexplained silence. Filed against the client (`UPSTREAM-claude-code.md`), not here — **ast-grep did its part.**

- **❌ RETRACTED: "on reload, ast-grep silently ignores an invalid rule (#722)".** We wrote this into the README, the CHANGELOG, a source comment and this file, on the strength of an issue **title**. **It is false on current versions, and the truth inverts the blame.** Reproduced against 0.44.1: an invalid rule on reload emits **both** `window/showMessage` \[ERROR] *and* `window/logMessage` \[ERROR] — `Failed to load rules: Cannot parse rule …` — and the last-known-good ruleset stays in effect (valid rules keep firing). On *startup* the same two errors fire and zero rules load. Not silent by any reading. #722's own reproduction (a missing `id:` field) no longer reproduces either: ast-grep now derives the id from the filename.
  #722 is still open, but stale (last activity 2024-09), and the maintainer's reply — *"It sounds like an LSP client feature to me"* — **was right**: ast-grep reports the failure, and it is the client that must surface it. The observed silence in Claude Code is the client discarding `window/showMessage` and `window/logMessage`. **We nearly filed a bug against ast-grep for a defect that was the client's, and that ast-grep had already done its half of.** Recorded because it is the single most instructive mistake in this whole investigation: a bug report is not a behaviour spec, and an issue title is not evidence.

## Upstream Opportunities

- **An ast-grep language-server plugin for Claude Code** (2026-07-11) — Claude Code supports registering language servers from a plugin (`lspServers`), and `ast-grep lsp` works well with it: rules produce live diagnostics in-context after each edit. No such plugin existed, so we built one — including the extension-collision rule that makes it safe to install (never claim a file extension an official language server owns, because the first server registered for an extension wins and the others never start), and a zero-dependency stdio LSP verifier that asserts a diagnostic **fires** on a violating buffer and **clears** on the fixed one.
  Might suit ast-grep's own docs (`guide/tools/editors.html` lists nvim/coc/helix/emacs but not Claude Code), or its existing Claude Code plugin (see `UPSTREAM-plugin--ast-grep.md`).
  Source: `voxpelli/claude-astgrep` (this repo) · Merge readiness: direct
  Ownership: us · Workaround: full — published as a standalone plugin.
