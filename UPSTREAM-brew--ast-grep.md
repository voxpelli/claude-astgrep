## Feature Requests

- **`ast-grep lsp` cannot watch its own rule files ‚Äî hot-reload dies on any client without file watching** (2026-07-11) ‚Äî The language server does not watch anything itself. On `initialized` it sends a `client/registerCapability` request asking the **client** to watch `**/*.{yml,yaml}` and report back via `workspace/didChangeWatchedFiles`. That is legitimate LSP, and it works in VSCode. But it makes hot-reload entirely **client-dependent**: a client that does not implement file watching (Claude Code does not) silently never reloads rules, and the server serves its startup rule set forever.
  The failure is invisible from both ends. The server behaves perfectly ‚Äî document sync, re-analysis, diagnostics all correct ‚Äî and simply never learns the rules changed. The user edits a rule, sees no change, and concludes their *rule* is wrong. Confirmed the reload path itself is fine: injecting `workspace/didChangeWatchedFiles` manually over stdio makes the server log "Configuration files changed, reloading rules‚Ä¶ Rules reloaded successfully".
  **THE DURABLE FIX, and it is entirely server-side: a self-watch fallback when the client does not advertise `workspace.didChangeWatchedFiles.dynamicRegistration`.** It needs nothing from Anthropic, and it would retire our shim completely.
  **Two measurements make the case, both taken 2026-07-11:**
  1. **Claude Code advertises `dynamicRegistration = undefined`** ‚Äî it does not claim to watch files, which is honest and correct for a client that cannot. So the signal a fallback would key on **is already present and is currently ignored**. (Measured by logging the client's `initialize` from inside the server, via `claude --plugin-dir ‚Ä¶ --debug-file`. No public dump of Claude Code's capabilities existed before this.)
  2. **ast-grep registers unconditionally anyway**, without checking that capability (see the Bugs section). So today it asks a client that has openly said it cannot watch files to please watch files, and then trusts the silence.
  **Strong precedent:** **rust-analyzer** ships exactly this fallback (`files.watcher: "client" | "server"`, server-side via `notify`), and **pyright** runs its own watchers. Two major servers already concluded client-provided watching cannot be relied on ‚Äî and both are consequently *immune* to this whole class of failure in Claude Code, while ast-grep is not.
  **The counter-argument to expect:** #2142 (closed, shipped 0.39.4) is the maintainer explicitly choosing the current design ‚Äî *"I found LSP should ask Client to provide file watch and reload config. So this should be done in this repo instead of vscode or other clients."* That is a fair call **for a spec-compliant client**. The ask here is not to reverse it, but to add the fallback for clients that decline the capability ‚Äî which is precisely what the capability exists to signal.
  Interacts badly with **#722** (open, *"LSP: inform user when rule is invalid"*): on reload an invalid rule is silently ignored, so a torn read of a half-written file makes diagnostics quietly vanish rather than error.
  Ownership: upstream ¬∑ Workaround: **full** ‚Äî `bin/lsp-shim.mjs` in this repo (0.4.0) answers the `registerCapability` request on the client's behalf, watches the globs the server registered, and injects `workspace/didChangeWatchedFiles`. Verified end-to-end by `verify-lsp-reload.mjs`.

- **`ast-grep lsp` hard-exits when no `sgconfig.yml` is present** (2026-07-11) \[minor] ‚Äî The server exits during `initialize` with `Error: No ast-grep project configuration is found.` (exit code 3). Reasonable in isolation, but it means an editor integration that registers the server **globally** gets a dead server in every project that does not use ast-grep ‚Äî which is most of them. The VSCode extension avoids this by gating activation on `findConfigFile()`; a client whose plugin model has no activation condition (Claude Code) cannot.
  Worse, in a client where the first server claiming a file extension wins and the others never start, a dead-on-arrival ast-grep server could, depending on unspecified client behaviour, *starve* that extension for the real language server. That risk forces a defensively narrow extension claim.
  A `--no-config-ok` / no-op mode (start, report zero rules, publish nothing) would let integrations register the server broadly and safely.
  Severity: minor ¬∑ Ownership: upstream ¬∑ Workaround: full ‚Äî pass `-c <path>` explicitly and document that the plugin is a no-op without a config.

## Bugs

- **`ast-grep lsp` registers `didChangeWatchedFiles` dynamically without checking whether the client supports it** (2026-07-11) \[minor] ‚Äî LSP requires a server to consult `ClientCapabilities` before using a dynamic registration: `workspace.didChangeWatchedFiles.dynamicRegistration` exists precisely so a client can say *"do not do this to me."* ast-grep sends `client/registerCapability` on `initialized` **unconditionally**. Measured directly: a client whose `initialize` carries `capabilities: {}` still receives the registration for `**/*.{yml,yaml}`.
  Impact is close to nil today, and the honest framing is that **this bug is currently load-bearing for everyone stuck behind a client that ignores the registration** ‚Äî it is the only reason a stdio shim can intercept anything. So it should not be "fixed" in isolation: gating on the capability without also adding the **self-watch fallback** requested above would take working hot-reload *away* from those clients and give nothing back. The two changes are one change.
  A capability check is also the natural trigger for that fallback: client says it can watch ‚Üí register, as today; client is silent ‚Üí watch in-process. That is what rust-analyzer does.
  Severity: minor ¬∑ Ownership: upstream ¬∑ Workaround: full ‚Äî our shim injects `dynamicRegistration: true` into the capabilities it forwards, so it is correct whether or not ast-grep starts checking. See `bin/lsp-shim.mjs`.

- **The server ignores the `languageId` the client sends and re-detects the language from the file path** (2026-07-11) \[minor] ‚Äî A client that opens a buffer with `languageId: "javascript"` at a path ast-grep maps differently gets ast-grep's answer, not the client's. Only the *keys* of a `extensionToLanguage`-style map therefore matter; the values are cosmetic.
  Mostly benign, and arguably pragmatic, but it means the server cannot lint an unsaved/virtual buffer whose language is known only from `languageId`, and it silently diverges from every other language server on a detail clients are entitled to rely on. Worth documenting even if not changed.
  Severity: minor ¬∑ Ownership: upstream ¬∑ Workaround: full ‚Äî name fixture files with the extension the rule's `language:` implies.

- **The server hard-exits during `initialize` instead of returning an `InitializeError`** (2026-07-11) \[minor] ‚Äî With no `sgconfig.yml` it prints `Error: No ast-grep project configuration is found.` and exits (code **3**; an earlier note of ours said 6 ‚Äî measured, it is 3). LSP has a defined shape for this: fail the `initialize` request with an `InitializeError` and let the client decide whether to retry or report. Killing the process mid-handshake gives the client nothing structured to show a user, and in a client that registers servers globally it means a dead server in every project that does not use ast-grep.
  Severity: minor ¬∑ Ownership: upstream ¬∑ Workaround: full ‚Äî pass `-c <path>` explicitly and document that the plugin is a no-op without a config.

## Upstream Opportunities

- **An ast-grep language-server plugin for Claude Code** (2026-07-11) ‚Äî Claude Code supports registering language servers from a plugin (`lspServers` in `plugin.json`), and `ast-grep lsp` works with it: rules produce live diagnostics in-context after each edit. No such plugin existed. Built one ‚Äî including the collision rule that makes it safe to install (never claim a file extension owned by an official language server, because the first server registered for an extension wins and the others never start), plus a zero-dependency stdio LSP verifier that asserts a diagnostic **fires** on a violating buffer and **clears** on the fixed one.
  Would suit either ast-grep's own docs (`guide/tools/editors.html` lists nvim/coc/helix/emacs but not Claude Code) or its existing Claude Code plugin (see `UPSTREAM-plugin--ast-grep.md`).
  Source: `voxpelli/claude-astgrep` (this repo) ¬∑ Merge readiness: direct
  Ownership: us ¬∑ Workaround: full ‚Äî published as a standalone plugin.
